# Dynamic Permission System Implementation

## Overview
Triển khai hệ thống Role-Based Access Control (RBAC) dynamic với các tính năng:
- Quản lý vai trò (role) và quyền (permission) linh hoạt
- Kiểm soát truy cập tài nguyên cụ thể (exam, problem, submission, etc.)
- Audit log cho tất cả thay đổi quyền
- Middleware để enforce permissions trên các endpoint

## Database Schema

### Các bảng tạo mới (005_create_permissions.sql)

1. **roles**
   - id, name, description, is_extensible, created_at, updated_at
   - Seed data: admin, lecturer, student

2. **permissions**
   - id, resource_type, action, description, created_at
   - Resource types: exam, problem, submission, user, role, permission, class, report
   - Actions: add, view, update, delete, import, export

3. **role_permissions** (Many-to-many)
   - Maps roles to permissions
   - Pre-populated with role-specific permission sets

4. **user_roles** (Many-to-many)
   - Maps users to roles
   - Tracks who assigned the role (assigned_by)

5. **resource_access_control**
   - Tracks ownership and access levels for specific resources
   - resource_type, resource_id, user_id, permission_type (owner/editor/viewer)
   - For lecturer ownership of exams/problems, student enrollment to classes

6. **permission_audit_log**
   - Logs all permission/role changes for compliance
   - Tracks: action, target_user_id, target_role_id, performed_by, details, created_at

## Permission Matrix

### Admin Role
- **Permissions**: All (selected via `WHERE r.name = 'admin'`)
- Access: Full system access

### Lecturer Role
- **Resources**: exam (all CRUD), problem (all CRUD), class (all CRUD), submission (view, export), report (view)
- Access: Manage own exams/problems, view student submissions

### Student Role
- **Resources**: exam (view), submission (add, view), problem (view)
- Access: Take enrolled exams, submit solutions, view own submissions

## Admin Endpoints

### Base: `/admin` (requires auth + admin role middleware)

#### System Management
- `GET /admin/stats` - Get system statistics
- `GET /admin/roles` - List all roles
- `GET /admin/permissions` - List all permissions

#### User Management
- `GET /admin/users?page=1&pageSize=20` - List users (paginated)
- `POST /admin/users/import` - Import users from JSON
- `PUT /admin/users/{id}` - Update user details
- `PUT /admin/users/{id}/role` - Update user role (legacy)
- `PUT /admin/users/{id}/active` - Toggle user active status

#### Role Assignment
- `POST /admin/users/{userId}/roles` - Grant role to user
  - Request: `{roleId: int32}`
- `DELETE /admin/users/{userId}/roles` - Revoke role from user
  - Request: `{roleId: int32}`
- `GET /admin/users/{userId}/roles` - Get user roles

#### Permission Management
- `GET /admin/roles/{roleId}/permissions` - Get role permissions
- `POST /admin/roles/{roleId}/permissions` - Grant permission to role
  - Request: `{permissionId: int32}`
- `DELETE /admin/roles/{roleId}/permissions/{permissionId}` - Revoke permission

#### Audit
- `GET /admin/audit-log?page=1&pageSize=20` - Get audit log (paginated)

## Key Design Decisions

### 1. Resource Ownership Tracking
- Lecturers own exams/problems they create
- Stored in `resource_access_control` table with permission_type='owner'
- Enables "lecturer can only manage own resources" pattern

### 2. Permission Hierarchy
- Role-based permissions first (via user_roles → role_permissions)
- Resource-specific permissions second (via resource_access_control)
- Combined check: user must have BOTH role permission AND resource access

### 3. Flexible Extensions
- Permission matrix is pre-seeded but system is flexible
- Can add new resource types and actions without code changes
- Roles can be extended if is_extensible=true

### 4. Audit Trail
- All permission/role changes logged
- Tracks who performed the action (performed_by)
- Optional details field for additional context (JSON)

## Usage Examples

### Adding permission middleware to exam creation endpoint
```go
router.POST("/exams", 
    authMiddleware,
    permissionMiddleware.PermissionMiddleware(permService, "exam", "add"),
    handler.CreateExam,
)
```

### Adding resource permission check for exam update
```go
router.PUT("/exams/:id",
    authMiddleware,
    permissionMiddleware.ResourcePermissionMiddleware(permService, "exam", "update"),
    handler.UpdateExam,
)
```

### Owner or admin check for delete
```go
router.DELETE("/exams/:id",
    authMiddleware,
    permissionMiddleware.OwnerOrAdminMiddleware(permService, "exam"),
    handler.DeleteExam,
)
```
